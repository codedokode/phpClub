Программирование включает в себя, в том числе, умение правильно оформлять код. На неакуратно написанный код не хочется даже смотреть.

Если тебе лень выравнивать код руками, закачай его на http://beta.phpformatter.com/ и нажми «format». Что может быть проще?

Вот, как оформляем код мы:

Правила основаны на Zend Coding Guides (http://framework.zend.com/manual/1.12/en/coding-standard.html — на англ. яз.), основные моменты:

- мы используем для отступов 4 пробела
- в именах переменных и функций используется camelCase, первая буква почти всегда маленькая, знак подчеркивания «_» не используется. Название функции начинается с глагола, «сделайЧтоТо». Примеры: $numberOfPeople, printResults()
- в именах классов используется camelCase, первая буква большая, «_» может использоваться
- мы предпочитаеи подстановку переменных вместо конкатенации строк: "I am $age years old" — хорошо, 'I am ' . $age . ' years old' — плохо
- мы не занимаемся микрооптимизациями и заменой двойных кавычек на одинарные
- у управляющих кострукций скобки ставятся так:


if (...) {
 // ... 
} else {
 // ...
}


- у определений функций и классов так:


function makeMeFeelGood()
{
 // ...
}


В общем, пиши как на картинке, и проблем не будет.

Люди, воспитанные на Ява/C++ просто не способны понять все его преимущества». По моему мнению, в яваскрипте просто нет нормального ООП, а есть примитивные объекты-словари и костыли к ним (список костылей: http://javascript.crockford.com/inheritance.html , http://javascript.crockford.com/prototypal.html ). Подобные костыли включены в фреймворки и библиотеки.

Каждый раз, как я хочу написать что-то объектно-ориентиронное на JS, мне хочется плеваться. Действительно, видно, что язык написан за 8 дней и предназначен для проверки введенных в форму данных, а не разработки приложений. В названии есть слово «Java», но это выглядит как издевательство.

Итак, проблемы в JS:

1) объекты (и массивы), созданные на прототипе, расшариваются между всеми экземплярами класса:


function Cat(name) { this.name = name; }

Cat.prototype.children = []; // дети кошки
Cat.prototype.addChild = function (child) {
 this.children.push(child);
 return this;
}

var white = new Cat('White Cat');
var black = new Cat('Black Cat');
var whiteSon = new Cat('Whiteson');
var blackSon = new Cat('blackson');
white.addChild(whiteSon);
black.addChild(blackSon);
console.log(white.children); // выведет [Cat, Cat] — массив children общий у всех кошек



Выход — массив придется создавать в конструкторе, в то время как в нормальных языках он указывается в описании класса:


function Cat(name) {
 this.name = name;
 this.children = [];
}


Веселее все становится, если мы хотим наследоваться. Тут нас ждут ловушки:

2) для вызова родительского метода надо писать уродливые конструкции

Parent.prototype.method.call(this, arg1, arg2);

вместо

parent::method($arg1, $arg2) в PHP.

3) конструктор не может ничего делать, кроме создания свойств. Пример, допустим мы хотим написать класс для превращения input[type=text] в счетчик:


function Counter(input, initialValue) {
 this.input = input;
 this.value = initialValue;
 input.value = initialValue;
}
Counter.prototype.increment = function () {
 this.value ++;
 this.input.value = this.value;
};

// Попробуем унаследовать класс SuperCounter, который умеет еще и уменьшать значения:
function SuperCounter() {
 Counter.apply(this, arguments);
}

// Но унаследовать мы не сможем, так как эта строка даст: 
// TypeError: Cannot set property 'value' of undefined
SuperCounter.prototype = new Counter();

// Надо починить свойство constructor, само оно не починится
SuperCounter.prototype.constructor = SuperCounter;

SuperCounter.prototype.decrement = function () {
 this.value --;
 this.input.value = this.value;
};



Нормальное решение этой проблемы я так и не придумал. Решение, естественно, должно корректно работать с instanceof. Стоит ли говорить, что в языках вроде PHP такой проблемы просто напросто нет?

4) instanceof не работает, если вы используете фреймы. x instanceof Array не работает, cat instanceof Cat не работает, если x и cat созданы в другом фрейме. Ок, вы сами себе злобный буратино, если до сих пор используете фреймы, но все равно, это плохо.

5) Type hints в яваскрипте, естественно, тоже нет. Пишите ручками в каждой функции assert(x instanceof Class) или что похуже. В PHP они есть.

6) Яваскрипт позволяет на ходу добавлять к объекты свойства и методы. Замечательно, опечатался в имени свойства, и вместо жуткого эксепшена (или ошибки компиляции) программа молча создает новое свойство и продолжет работать, как ни в чем не бывало, потеряв данные. Причем эта возможность динамически создавать свойства, опять же, подается как преимущество перед языками вроде Ява. Ага, да, преимущество, которое позволяет вам проводить больше времени за отладкой в любимой колсоли разработчика.

function fn () { return 1234; }

echo fn();

$x = fn(); // в x будет 1234

использовать в выражении:

echo fn() + fn(); // выведет 2468

или даже передать в другую функцию:

echo sqrt(fn()); // выведет 35.128336140501 — корень из числа 1234 (функция sqrt входит в состав php и вычисляет квадратный корень).
